Six RC-blocking findings surfaced during Phase 4 BQA. Findings 1, 2, 5, and 6 are isolated template/mail class corrections. Findings 3 and 4 share one root cause in the delivery service and are fixed together.

## Root causes

**Findings 1, 2, 5, 6 — Mail template and header defects**

- **Finding 1:** `invoice-paid.blade.php` renders the TXID as plain text with no word-break treatment. Bitcoin TXIDs are 64-character hex strings that overflow their container on narrow viewports. Mail clients require inline styles — class-based rules are stripped.
- **Finding 2:** Both past-due templates label the outstanding USD balance as approximate, but `number_format($outstandingUsd, 2)` produces a to-the-cent figure. The qualifier is false.
- **Finding 5:** All outbound client-facing mail uses `no-reply@cryptozing.app` as the from address and sets no `Reply-To` header. The issuer owns the client relationship — when a client replies to an invoice, receipt, or past-due notice, that reply should reach the issuer, not a dead address. Fix: set `Reply-To` to the issuer's email on all client-facing Mailable classes. The `From` address stays `no-reply@cryptozing.app`.
- **Finding 6:** `invoice-ready.blade.php` renders `{{ strtoupper($invoice->status ?? 'draft') }}` directly to the client. "DRAFT" and "SENT" are internal issuer terms. The client-facing label for an invoice awaiting payment is "Open."

**Findings 3 & 4 — Past-due delivery log bloat**

`InvoiceAlertService::sendPastDueAlerts()` iterates all three sequence slots on every cron run. Inside each iteration it calls `InvoiceDeliveryService::queue()`, and `queue()` always calls `createDelivery()`, which always writes a new `InvoiceDelivery` row — even when the result is `skipped`. The loop's `break` only fires when a slot produces a newly-queued delivery. Once all three slots have been sent, every subsequent daily cron run completes the full iteration and writes 2–6 skipped rows per invoice (issuer + client × up to 3 slots). An invoice past-due for six months accumulates ~1,080 rows; a year produces thousands. This is also why the delivery log appears to show constant activity — no extra mail is being sent, but the log fills with noise.

## 3. Receipt resend

Issuers have no way to resend a receipt once one has been sent. The existing `queue()` idempotency guard blocks re-sends for `receipt` type once any `sent` row exists for that invoice + recipient + null context key. A resend is an explicit issuer action and should be treated as a distinct intent.

**Design decisions:**
- `InvoiceDeliveryService::queueResend()` owns the resend path. It generates a `resend_{uuid}` context key internally, making each resend intent unique to the idempotency guard without modifying `queue()`.
- Cooldown: before queuing, check for any recent receipt delivery (any context key) within `manual_send_cooldown_minutes`. This prevents spam-clicking. The check is in `queueResend()` so the cooldown is enforced consistently regardless of entry point.
- UI: a "Resend receipt" button on the invoice show page, visible only when a receipt has already been sent (`status` in `['queued', 'sending', 'sent']`).
- After resend: redirect back to the invoice show page with a success or throttled flash message.

1. [ ] [Agent] Add `queueResend(Invoice $invoice, string $type, string $recipient): InvoiceDelivery` to `InvoiceDeliveryService`. Internally generates context key `'resend_' . Str::uuid()`. Before queuing, checks for any recent delivery of the same type on the same invoice (regardless of context key) within `manualSendCooldownMinutes()` — if found, returns a skipped delivery with a throttle reason. Otherwise delegates to `queue()` with the generated context key.
2. [ ] [Agent] Add a `resendReceipt` action to `InvoiceDeliveryController` (POST). Gate: invoice must be paid, must belong to the authenticated user, and must have at least one sent receipt delivery.
