## Finding 1 — Receipt TXID wraps poorly on narrow screens

**Root cause:** `resources/views/mail/invoice-paid.blade.php` renders the TXID as plain text inside a `<x-mail::panel>` block with no word-break treatment. Bitcoin TXIDs are 64-character hex strings. Mail client table cells do not break arbitrary strings by default, so the string overflows its container on narrow viewports.

**Fix:** Wrap the TXID in an inline-styled `<span>` with `word-break: break-all` applied directly. Inline styles are required because mail clients strip `<style>` blocks and class-based rules.

### Implementation

1. [ ] In `resources/views/mail/invoice-paid.blade.php`, locate the TXID line inside the panel block and replace it with:
   ```
   **TXID:** <span style="word-break: break-all; font-family: monospace;">{{ $invoice->txid ?? '—' }}</span>
   ```

### Verification

1. [ ] Send a receipt email for a real paid invoice (use the test environment).
2. [ ] Open the email in a desktop mail client at full width — confirm the TXID is visible and does not overflow the panel.
3. [ ] Resize the browser or view the email on a mobile device (or use the Mailgun "preview" feature if available) — confirm the TXID wraps within the panel at narrow widths rather than overflowing horizontally.
4. [ ] Confirm the TXID value itself is unchanged (no truncation, the full hash is visible).

## Finding 2 — Past-due notice USD value is falsely marked approximate

**Root cause:** Both past-due templates label the outstanding USD balance as approximate, but `number_format($outstandingUsd, 2)` produces a to-the-cent figure. "Approximately" is false and erodes trust.

- `resources/views/mail/invoice-past-due-client.blade.php`: `approximately **${{ number_format($outstandingUsd, 2) }} USD**`
- `resources/views/mail/invoice-past-due-issuer.blade.php`: `**Outstanding:** ${{ number_format($outstandingUsd, 2) }} (approx.)`

**Fix:** Remove the approximate qualifier from both templates. The computed value is exact enough to present as-is.

### Implementation

1. [ ] In `invoice-past-due-client.blade.php`, change:
   ```
   approximately **${{ number_format($outstandingUsd, 2) }} USD**
   ```
   to:
   ```
   **${{ number_format($outstandingUsd, 2) }} USD**
   ```

2. [ ] In `invoice-past-due-issuer.blade.php`, change:
   ```
   **Outstanding:** ${{ number_format($outstandingUsd, 2) }} (approx.)
   ```
   to:
   ```
   **Outstanding:** ${{ number_format($outstandingUsd, 2) }}
   ```

### Verification

1. [ ] Trigger a past-due alert for a test invoice (manually call `invoices:send-past-due-alerts` or use Tinker to queue the delivery directly).
2. [ ] Open the client past-due email — confirm the outstanding balance is shown without any "approximately" or "approx." prefix or suffix.
3. [ ] Open the issuer past-due email — confirm the outstanding balance is shown without "(approx.)".
4. [ ] Confirm both emails still show the correct to-the-cent figure.

## Findings 3 & 4 — Past-due notices fire too frequently / delivery log bloat

These two findings share one root cause and one fix.

**Root cause:** `InvoiceAlertService::sendPastDueAlerts()` iterates all 3 sequence slots on every cron run. Inside the loop, it calls `InvoiceDeliveryService::queue()` for each slot — and `queue()` always calls `createDelivery()`, which always writes a new `InvoiceDelivery` row, even when the result is 'skipped'. The `break` in the loop only fires when a slot is *newly* queued. Once all 3 slots have been sent, every subsequent daily cron run completes the full 3-iteration loop, writing 2–6 skipped rows per invoice per day (issuer + client × up to 3 slots). An invoice that has been past-due for 6 months accumulates ~1,080 skipped rows; a year-old invoice can easily reach several thousand.

This is also why issuers perceive past-due notices as firing too frequently — the delivery log shows constant activity even though no mail is actually being sent. The visual noise in the log is the report.

**Fix:** In `sendPastDueAlerts()`, check whether a delivery already exists for each slot's context key before calling `queue()`. If both the issuer and client delivery rows already exist for a given slot (any status — queued, sent, skipped, failed), skip the `queue()` call entirely rather than creating another skipped row. Add a private helper to `InvoiceDeliveryService` that checks existence without creating a row.

### Implementation

1. [ ] Add `deliveryExists(Invoice $invoice, string $type, string $recipient, string $contextKey): bool` to `InvoiceDeliveryService`. This method uses `matchingDeliveries()` with `->exists()` and checks any status (not just sent or queued). It should NOT create any rows — it is read-only.

2. [ ] In `InvoiceAlertService::sendPastDueAlerts()`, before calling `queue()` for each recipient in a slot, call `deliveryExists()` first. If a row already exists for that type+recipient+contextKey combination, skip the `queue()` call for that recipient. Only call `queue()` when no prior row exists. Preserve the existing `newlyQueued` break logic.

3. [ ] Confirm the logic still correctly advances through slots when slot 1 is sent but slot 2 has not fired yet — the slot 2 check should find no existing row and proceed to `queue()` as before.

### Verification

1. [ ] Run `invoices:send-past-due-alerts` twice in a row against a test invoice that already has all 3 past-due slots sent. Confirm no new `InvoiceDelivery` rows are created on the second run.
2. [ ] Confirm that a fresh past-due invoice (slot 1 not yet sent) still produces a queued delivery on the first run.
3. [ ] Confirm that slot 2 fires correctly once the invoice is 7+ days past due and slot 1 is already sent.
4. [ ] Run the full test suite — confirm all existing delivery dedup tests still pass.
5. [ ] Inspect the delivery log for a production invoice with many existing skipped rows and confirm no new rows are being appended after the fix deploys.

## Finding 5 — Reply-to address is no-reply@cryptozing.app

**Root cause:** No Mailable class sets a `replyTo` header on its `Envelope`. The `MAIL_FROM_ADDRESS` env var is set to `no-reply@cryptozing.app`, so replies go there by default. That address is unmonitored. Several mail templates explicitly invite replies ("just reply to this email") — those replies currently go nowhere.

**Fix:** Add a `MAIL_REPLY_TO_ADDRESS` / `MAIL_REPLY_TO_NAME` env var pair. Set a monitored address there (e.g. `support@cryptozing.app` or the catch-all). Apply it globally via a shared base Mailable or by setting the `reply_to` key in `config/mail.php` as an app-level default. All outbound mail will pick it up automatically.

Setting it in `config/mail.php` as a `reply_to` array (mirroring the existing `from` array) and applying it in a base Mailable or a service provider `Mailable::macro` is the cleanest approach — it avoids touching each of the 14 Mail classes individually.

### Implementation

1. [ ] Add `reply_to` to `config/mail.php`:
   ```php
   'reply_to' => [
       'address' => env('MAIL_REPLY_TO_ADDRESS', env('MAIL_FROM_ADDRESS', '')),
       'name'    => env('MAIL_REPLY_TO_NAME', env('MAIL_FROM_NAME', '')),
   ],
   ```

2. [ ] Create `app/Mail/BaseMail.php` as an abstract base class extending `Illuminate\Mail\Mailable`. Override `envelope()` to call the concrete class's envelope and merge the configured `reply_to` into it, OR use a `boot` / `resolving` hook in a service provider to apply the reply-to to all Mailable instances before send. Choose whichever approach does not require touching each of the 14 Mail classes.

   **Preferred approach:** Add a `Mailable::buildUsing` or `resolving` hook in `AppServiceProvider::boot()`:
   ```php
   \Illuminate\Mail\Mailable::macro('withReplyTo', function () {
       $replyTo = config('mail.reply_to');
       if (!empty($replyTo['address'])) {
           $this->replyTo($replyTo['address'], $replyTo['name'] ?: null);
       }
       return $this;
   });
   ```
   Then hook it via the mail manager's `sending` event or use the `Mailable` `boot` pattern. Investigate the cleanest Laravel 11 hook point and use it.

3. [ ] Add `MAIL_REPLY_TO_ADDRESS=support@cryptozing.app` to `.env.example` with a comment explaining it should be a monitored address.

4. [ ] Confirm `.env` (local) has the reply-to set to the local catch-all so test mail routes correctly.

### Verification

1. [ ] Send a test invoice delivery email. Check the raw email headers (via Mailgun event log or the mail preview in the local mailer) — confirm a `Reply-To:` header is present and points to the configured address, not `no-reply@cryptozing.app`.
2. [ ] Repeat for at least one other mail type (e.g. receipt, past-due client notice) — confirm Reply-To is consistent across mail types.
3. [ ] Confirm the `From:` address is unchanged (still `no-reply@cryptozing.app` or whatever `MAIL_FROM_ADDRESS` is set to).

## Finding 6 — Invoice delivery email shows "Draft" status

**Root cause:** `resources/views/mail/invoice-ready.blade.php` renders `{{ strtoupper($invoice->status ?? 'draft') }}` in the panel, which outputs the internal status string (`DRAFT`, `SENT`, etc.) directly to the client. "Draft" is an internal issuer concept that has no meaning to a recipient who just received the invoice. "Sent" is similarly issuer-centric.

**Fix:** Replace the dynamic status expression with the client-facing label `Open`. Per the decision made in Phase 5: invoices awaiting payment are "Open" from the client's perspective. "Due" is reserved for date-related contexts.

### Implementation

1. [ ] In `resources/views/mail/invoice-ready.blade.php`, replace:
   ```
   **Status:** {{ strtoupper($invoice->status ?? 'draft') }}
   ```
   with:
   ```
   **Status:** Open
   ```

### Verification

1. [ ] Send a test invoice delivery email via the invoice show page.
2. [ ] Open the email and confirm the panel shows `Status: Open` — not `DRAFT`, `SENT`, or any other internal status string.
3. [ ] Confirm all other panel fields (due date, BTC amount) are unchanged.
