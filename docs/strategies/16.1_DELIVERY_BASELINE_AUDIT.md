# MS16 Phase 1 Strategy - Delivery Baseline Audit

Status: Active.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md), current delivery/alert code under `app/` and `tests/`

Phase 1 locks the trustworthy outbound-mail baseline for MS16 before any provider recovery work or broader notification-surface polish.

## 1. Reconstruct the live outbound surface
1. [ ] Inventory every delivery type the app can currently queue and map each one to one of; use the live code, not older milestone notes, as the baseline inventory:
   - canonical in spec/product scope
   - still in RC scope but not yet fully specified
   - stale drift to retire or reshape
   - Parallel sidecar that can run immediately while this inventory is in progress: inventory mailable classes, delivery types, and tests under `app/Mail`, `app/Jobs`, `app/Services`, and `tests/Feature/*Notification*`.
2. [ ] Capture the current outbound surface explicitly:
   - manual invoice send (`send`)
   - paid receipt (`receipt`)
   - owner paid notice (`owner_paid_notice`)
   - past-due owner/client alerts (`past_due_owner`, `past_due_client`)
   - overpayment owner/client alerts (`owner_overpay_alert`, `client_overpay_alert`)
   - underpayment owner/client alerts (`owner_underpay_alert`, `client_underpay_alert`)
   - partial-payment warnings (`owner_partial_warning`, `client_partial_warning`)

## 2. Identify the stop-the-bleed risks
1. [ ] Trace where deliveries are created, deduped, cooled down, skipped, and retried today.
2. [ ] Identify the runaway/spam-prone path concretely enough to say:
   - what can re-queue unexpectedly
   - which notice classes are currently weakest
   - whether the risk is manual-send, watcher-triggered, scheduler-triggered, queue-behavior, provider-behavior, or a combination
3. [ ] Document which current safeguards already exist and which are too narrow:
   - one-off `exists()` checks on receipts / owner paid notices
   - per-invoice timestamp cooldowns for past-due, overpay, underpay, and partial-warning alerts
   - queued-delivery skip logic when invoice truth changes before send
   - Parallel sidecar once this safeguard audit is underway: collect a concise matrix of current cooldown fields and skip logic from `app/Models/Invoice.php`, `app/Services/InvoiceAlertService.php`, and correction-related tests without changing scope.
4. [ ] Flag current drift that later phases must resolve instead of inheriting silently:
   - manual sends currently have no shared cooldown/dedupe guard
   - current partial-payment warning mail is broader than the trimmed notifications spec
   - issuer-copy behavior is inconsistent across current flows and settings

## 3. Lock the Phase 2 and Phase 3 inputs
1. [ ] Produce the minimum safeguard list that later work must implement before broader mail polish:
   - shared dedupe / notification-intent gating
   - per-class cooldown policy
   - circuit-breaker or send-disable behavior
   - structured evidence for delivery failures and provider pressure
2. [ ] Produce the provider-recovery inputs needed for Phase 2:
   - what evidence is needed from Mailgun/sendability state
   - whether aliasing currently hides or distorts that evidence
   - what blocking constraints, if any, would prevent adopting Mailgun HTTP API in implementation
3. [ ] Produce the truthfulness/rationalization inputs needed for Phase 3:
   - whether repeated partial-payment warnings survive, narrow, or retire
   - whether owner copies remain explicit mail classes, become issuer-copy behavior, or split by notice class
   - which payment-triggered sends may need owner review after ambiguity/correction scenarios

Do not treat provider-account investigation as a Phase 1 sidecar if the next blocking decision depends on it; that work belongs to Phase 2.

## 4. Verify Phase 1
1. [ ] The live outbound inventory is written down in this phase doc and no longer lives as stale “candidate” notes in the milestone doc.
2. [ ] The runaway/spam-prone risk is described concretely enough to drive implementation.
3. [ ] Phase 2 and Phase 3 each have clear inputs rather than inheriting placeholder work.
