# MS17 Phase 3 Strategy - Off-Chain Payment Recording

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`docs/specs/MANUAL_PAYMENTS.md`](../specs/MANUAL_PAYMENTS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

Issuers occasionally receive payment by wire transfer, cash, check, or other off-chain methods. Without a way to record these, the ledger shows a false outstanding balance and the invoice is stuck. This phase adds an issuer-facing "Record manual payment" form alongside the existing payment correction tools so invoices can reach paid status without on-chain workarounds.

## 1. Data layer

1. [ ] Add a `is_manual` boolean flag (default false) to `invoice_payments`. Manual payments have no `txid` and no on-chain context.
2. [ ] Add `reference` (nullable string) and `received_at` (nullable date) columns to `invoice_payments` for manual payment metadata.
3. [ ] Confirm the payment ledger recalculation already handles manual payments correctly once the row exists, or adjust as needed.
4. [ ] Manual payments must not be voided, ignored, or reattributed — ensure correction tools guard against operating on manual rows.

## 2. Controller and form

1. [ ] Add a `StoreManualPayment` controller (or action on the existing `InvoicePaymentAdjustmentController`) under the authenticated invoice routes.
2. [ ] Form fields: amount (USD, required), reference/note (optional, e.g. "Wire transfer ref #12345"), date received (optional, defaults to today).
3. [ ] Validate: amount must be positive; invoice must not be voided; issuer must own the invoice.
4. [ ] On save: create the `InvoicePayment` row with `is_manual = true`, recalculate the ledger, and transition the invoice to `paid` if the balance is fully closed.
5. [ ] Add the route to `web.php` under the authenticated invoice group.

## 3. UI

1. [ ] Add the "Record manual payment" form to the invoice payment history section, alongside the existing small-balance resolver and reattribution tools.
2. [ ] Manual payment rows in the payment history table must be visually distinct from on-chain rows — label them clearly (e.g. "Manual payment") and omit txid/confirmation columns.
3. [ ] If the manual payment closes the balance, the receipt panel must become available as with any other paid state.
4. [ ] Do not show the form on voided invoices.

## 4. Verify Phase 3

1. [ ] A manual payment can be recorded on an open invoice; the ledger updates correctly.
2. [ ] A manual payment that closes the balance transitions the invoice to `paid` and makes the receipt panel available.
3. [ ] Manual payment rows are visually distinct in the payment history table.
4. [ ] The form is not available on voided invoices.
5. [ ] Existing correction tools (ignore, reattribute, small-balance resolver) do not operate on manual payment rows.
6. [ ] Feature tests cover the happy path and the voided-invoice guard.
