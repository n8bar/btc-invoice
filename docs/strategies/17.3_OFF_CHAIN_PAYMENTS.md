# MS17 Phase 3 Strategy - Off-Chain Payment Recording

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`docs/specs/MANUAL_PAYMENTS.md`](../specs/MANUAL_PAYMENTS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

Issuers occasionally receive payment by wire transfer, cash, check, or other off-chain methods. Without a way to record these, the ledger shows a false outstanding balance and the invoice is stuck. This phase adds an issuer-facing "Record off-chain payment" form alongside the existing payment correction tools.

The data model already handles this. The existing adjustment pattern — `is_adjustment = true`, synthetic `txid` like `'manual-' . Str::uuid()`, existing `sats_received`/`fiat_amount`/`usd_rate` columns, `refreshPaymentLedger()` — is exactly the right mechanism. Off-chain payment recording is an extension of `InvoicePaymentAdjustmentController`, not a new system.

## 1. Controller

The `store` action in `InvoicePaymentAdjustmentController` is the right home. Add a dedicated `storeOffChain` action (or a new `OffChainPaymentController`) that:

1. [ ] Accepts: `amount_usd` (required, positive), `note` (optional, e.g. "Wire ref #12345"), `received_at` (optional date, defaults to today).
2. [ ] Looks up the current BTC rate to derive `sats_received` and `fiat_amount` — same as the existing `store` action.
3. [ ] Creates an `InvoicePayment` row with `is_adjustment = true`, a synthetic `txid` prefixed `offchain-`, and `confirmed_at` set to `received_at` (or now).
4. [ ] Calls `refreshAdjustmentTargets()` — identical post-write logic to existing adjustments.
5. [ ] Guards: invoice must not be voided; issuer must own the invoice; amount must be positive.
6. [ ] Add the route to `web.php` under the authenticated invoice group.

## 2. UI

1. [ ] Add a "Record off-chain payment" form to the invoice payment history section alongside the small-balance resolver and manual adjustment form.
2. [ ] Off-chain payment rows in the payment history table must be visually distinct from on-chain rows — label them (e.g. "Off-chain payment") and omit txid/confirmation columns.
3. [ ] Do not show the form on voided invoices.

## 3. Verify Phase 3

1. [ ] Recording an off-chain payment updates the ledger correctly via `refreshPaymentLedger()`.
2. [ ] An off-chain payment that closes the balance transitions the invoice to `paid` and makes the receipt panel available.
3. [ ] Off-chain payment rows are visually distinct from on-chain rows in the payment history table.
4. [ ] The form is not shown on voided invoices.
5. [ ] Feature tests cover the happy path and the voided-invoice guard.
