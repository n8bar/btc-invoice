# MS18 Phase 1 Strategy — CMS Selection & Staging Setup

Status: Active.
Parent milestone doc: [`docs/milestones/18_PRERELEASE_CONTENT_SEO.md`](../milestones/18_PRERELEASE_CONTENT_SEO.md)

**Goal:** Select a CMS and staging workflow as joint decisions, set both up, and confirm the full publish pipeline works end-to-end — so Phase 2 can go straight to writing without tooling uncertainty.

## 1. Define requirements
1. [x] Agree on a short feature set / requirements for both the CMS and staging workflow before listing candidates.

**Must-haves:**
1. Easy content authoring — HTML or Markdown, whichever the tool makes natural
2. GitHub Pages compatible — works with the existing Pages setup for now
3. Shared templating — one layout for consistent nav/header/footer across all pages
4. Per-page SEO metadata — title, description, and canonical controllable per article
5. Portable output — produces static files that work on GitHub Pages now and nginx in production at MS21

**Should-haves:**
6. Staging support — a way to preview content before it goes live; dev server (LAN-only) is the intended staging environment
7. Fast build — small edits and tweaks should not require a slow pipeline
8. Sitemap-friendly — easy to keep `sitemap.xml` current as content is added
9. Video embed support — pave the way for post-RC video content; full video production deferred until after RC1
10. Simple content authoring — add a new article without touching config files

## 2. Evaluate CMS and staging options together
1. [x] Note the MS21 production architecture so candidates can be evaluated with it in mind — accommodate rather than constrain:
   - Dev server runs Laravel via Sail (PHP built-in server in Docker); no nginx in the container. Production will be nginx + PHP-FPM.
   - At MS21 cutover, static content files will be served by nginx directly alongside the Laravel app — path-based split, same domain, no PHP involved for content routes.
   - The staging area built during this phase is intended to become the production content site at cutover — not throwaway infrastructure.
   - Post-RC staging options: new dev path, or a password-protected area on production — to be decided at MS21.
2. [ ] List candidate CMS options with their staging implications, evaluated against the requirements above.
3. [ ] List candidate staging approaches with their CMS implications.
4. [ ] Identify any combinations that are ruled out by the requirements or by each other.

## 3. Make the joint decision
1. [ ] Select a CMS. Document the choice and rationale.
2. [ ] Select a staging approach. Document the choice and rationale.
3. [ ] Confirm the two choices are compatible and reinforce each other.

## 4. CMS setup
1. [ ] Add CMS config file and base layout to `site/`.
2. [ ] Convert `site/index.html` to use the base layout.
3. [ ] Confirm the existing placeholder page still renders correctly after conversion.
4. [ ] Confirm GitHub Pages builds and serves correctly.

## 5. Staging setup
1. [ ] Implement the chosen staging workflow alongside the existing Sail/Docker dev stack.
2. [ ] Create a test content page and run it through the full staging workflow.
3. [ ] Confirm the test page is reviewable in staging before it touches the live site.
4. [ ] Confirm the test page goes live correctly when promoted.
5. [ ] Remove or archive the test page after verification.

## Exit Criteria
- [ ] CMS and staging approach selected and documented.
- [ ] CMS set up; existing placeholder page unaffected.
- [ ] Staging workflow set up and verified end-to-end alongside Sail.
- [ ] Phase 2 can begin writing content immediately without tooling work.
