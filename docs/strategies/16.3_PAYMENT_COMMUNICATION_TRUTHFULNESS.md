# MS16 Phase 3 Strategy - Payment Communication Truthfulness

Status: Pending.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)
Depends on: [`docs/strategies/16.2_SAFEGUARDS_PROVIDER_RECOVERY.md`](16.2_SAFEGUARDS_PROVIDER_RECOVERY.md)
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/specs/PAYMENT_CORRECTIONS.md`](../specs/PAYMENT_CORRECTIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

Phase 3 aligns payment-triggered communication with the now-trustworthy delivery path and resolves the remaining truthfulness and alert-surface questions.

## 1. Lock the acknowledgment versus receipt model
Do not split the core acknowledgment-versus-receipt decision across sidecars; it drives too many downstream surfaces.

1. [ ] Decide whether MS16 introduces an automatic low-information payment acknowledgment before any reviewed receipt. If that acknowledgment exists, keep the copy deliberately narrow and non-promissory. Current candidate wording carried forward for review:
   - `A Bitcoin payment of 0.00123456 BTC was detected.`
   - `No action is needed right now.`
   - `The invoice issuer has been notified to review it promptly.`
   - Do not ship the final line without the matching owner-side follow-through: a prompt, visible review/receipt call to action in the dashboard and invoice payment history.

## 2. Resolve later-payment ambiguity handling
1. [ ] Decide whether owner validation remains a narrow later-payment safeguard or becomes a broader rule for ambiguous payment-triggered communication; keep the sequencing explicit that this is MS16 follow-up work, not retroactive MS14 reattribution scope.
2. [ ] Explicitly decide which outbound classes it covers and which remain outside that gate:
   - higher-certainty receipt follow-up
   - low-information acknowledgment
   - manual sends
   - past-due reminders

## 3. Rationalize the live alert surface against the canonical spec
1. [ ] Decide whether repeated partial-payment warnings survive as:
   - a retained client-facing alert
   - an owner-side / operational signal only
   - a retired alert family folded into underpayment behavior
2. [ ] Decide how issuer copies work for client-facing notification emails:
   - explicit paired owner/client delivery classes
   - default issuer copy behavior with a toggle
   - a hybrid split by notice class where justified
3. [ ] Decide how delivery-history labels and rows should represent the final notice classes and outcomes.
   1. [ ] Parallel: clean up delivery-history labels and copy once the final notice-class map is decided.

## 4. Ship the supporting UX and settings
1. [ ] Add the owner-facing review/send surfaces required by the final payment communication model.
2. [ ] Add the issuer-copy toggle for client-facing notification emails with a default-on RC posture if that behavior survives Phase 3.
3. [ ] Keep the settings, delivery log, and invoice surfaces aligned with the final terminology and outcomes.
   1. [ ] Parallel: finish settings-surface and UX follow-through work once the acknowledgment/receipt model is locked.

## 5. Verify Phase 3
1. [ ] Payment-triggered communication is truthful under ambiguity, ignore, reattribution, and already-funded-invoice scenarios.
2. [ ] Any retained payment-triggered mail classes have explicit, documented reasons to exist.
3. [ ] Owner review/send paths exist wherever the final communication model promises them.
