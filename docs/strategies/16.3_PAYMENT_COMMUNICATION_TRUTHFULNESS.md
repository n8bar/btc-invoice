# MS16 Phase 3 Strategy - Payment Communication Truthfulness + Notification UX

Status: Active.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)

Phase 3 aligns payment-triggered communication with the now-trustworthy delivery path, resolves the remaining truthfulness and alert-surface questions, and owns the active notification UX/settings needed to make the model testable now.

## 1. Lock the acknowledgment versus receipt model
Do not split the core acknowledgment-versus-receipt decision across sidecars; it drives too many downstream surfaces.

1. [x] MS16 introduces an automatic low-information payment acknowledgment before any later client receipt. Keep the copy deliberately narrow and non-promissory. Approved direction:
   - `A Bitcoin payment of 0.00123456 BTC was detected.`
   - `No action is needed right now.`
   - `The invoice issuer has been notified to review it promptly.`
   - Do not ship the final line without the matching owner-side follow-through: a prompt, visible review/receipt call to action in the dashboard and invoice payment history.
   - Ambiguity does not block this low-information acknowledgment as long as the copy remains narrow and non-promissory.

## 2. Resolve receipt review and later-payment ambiguity handling
1. [x] Keep owner review as the rule for higher-certainty client receipt mail; do not treat later-payment ambiguity as the only case that requires review. Keep the sequencing explicit that this is MS16 follow-up work, not retroactive MS14 reattribution scope.
2. [x] Explicitly decide which outbound classes it covers and which remain outside that gate:
   1. [x] Client receipt / paid confirmation is issuer-reviewed/manual for RC; no automatic client receipt path should remain, and issuer sending after review remains allowed.
   2. [x] Low-information acknowledgment is not blocked as long as the copy stays narrow and non-promissory.
   3. [x] Manual sends are not blocked by this ambiguity gate.
   4. [x] Past-due reminders are not blocked by this ambiguity gate.

## 3. Rationalize the live alert surface against the canonical spec
1. [x] Retire repeated partial-payment warnings as a separate alert family and fold them into underpayment behavior.
   - The current repeated warnings come from later partial-payment detections while an invoice still remains underpaid and the cooldown allows another warning cycle.
   - Phase 3 should remove that separate client-facing warning family rather than preserving a repeated partial-payment-specific mail path.
2. [x] Use explicit paired owner/client delivery classes for client-facing notification emails in RC rather than a generic issuer-copy toggle or hybrid split.
3. [x] Decide how delivery-history labels and rows should represent the final notice classes and outcomes. Approved direction:
   - Use `Notice`, not `Type`, in the invoice delivery log.
   - Show concise labels instead of raw storage keys.
   - Keep paired owner/client audience markers explicit in the label, for example `Past-due reminder (client)` and `Underpayment alert (owner)`.
   - Current approved delivery-log labels:
     - `send` -> `Invoice email`
     - `receipt` -> `Receipt (client)`
     - `owner_paid_notice` -> `Paid notice (owner)`
     - `past_due_client` / `past_due_owner` -> `Past-due reminder (client|owner)`
     - `client_overpay_alert` / `owner_overpay_alert` -> `Overpayment alert (client|owner)`
     - `client_underpay_alert` / `owner_underpay_alert` -> `Underpayment alert (client|owner)`
     - `client_partial_warning` / `owner_partial_warning` -> `Partial payment warning (client|owner)` while those legacy rows still exist
   - Future payment-triggered follow-up should keep the acknowledgment-versus-receipt split visible in delivery history once the new rows ship, using labels such as `Payment acknowledgment (client)` for the narrow automatic notice and `Receipt (client)` for the higher-certainty follow-up.
   - Outcome labels should display as `Queued`, `Sending`, `Sent`, `Skipped`, and `Failed`.
   1. [x] Parallel: clean up delivery-history labels and copy once the final notice-class map is decided.

## 4. Ship the supporting notification UX and settings
1. [x] Add the owner-facing review/send surfaces required by the final payment communication model.
   - Phase 3 now includes a manual client-receipt send path from the paid invoice payment history plus a dashboard `Review receipt` prompt when a paid invoice still lacks a queued/sent client receipt.
   - The payment watcher now queues paired `payment_acknowledgment_client` / `payment_acknowledgment_owner` rows keyed to the detected `txid`, so the low-information acknowledgment is visible in delivery history and lands before any later reviewed receipt.
2. [x] Remove the lingering automatic client-receipt path and align supporting copy to the manual-review-only receipt rule.
   - The earlier ambiguity-hold implementation can be reused as owner review context where it helps, but it should no longer drive an automatic receipt attempt or a skipped automatic receipt row.
   1. [x] Stop the `InvoicePaid` listener from auto-queueing or auto-skipping client receipts; keep the owner paid notice intact.
   2. [x] Replace automatic-hold wording with owner-review context wording in invoice and settings surfaces.
   3. [x] Keep the invoice and dashboard review/send prompts honest about receipts always being manual.
3. [x] Refrain from adding an issuer-copy toggle in Phase 3; paired owner/client delivery classes are the approved RC direction unless a later milestone reopens that choice.
4. [x] Keep the delivery log, invoice surfaces, and any surviving notification settings aligned with the final terminology and outcomes.
   1. [x] Parallel: finish receipt-model copy cleanup once item 4.2 lands.
   2. [x] Make the straightforward paid-invoice receipt state visibly actionable without looking like an ambiguity warning block or hiding the `Send receipt` CTA below the fold.
   3. [x] Fix invoice-page anchor targeting and sticky-nav offset behavior so linked sections land in the intended visible position.
   4. [x] Strengthen the dashboard `Review receipt` affordance with a low-risk pending indicator while preserving the current pill treatment.
5. [ ] Ship the active mail branding/settings work needed for the current notification UX so Browser QA exercises the real mails instead of Laravel-default chrome.
   - Keep this KISS: shared CryptoZing-branded mail chrome and only the minimal current notification settings/copy needed for the live receipt/acknowledgment model.
   - Do not expand this into a broad template editor or arbitrary body-copy customization surface in MS16.
   1. [x] Replace Laravel-default shared mail chrome with a simple CryptoZing-branded shared layout/header/footer/theme for the active Phase 3 mail classes.
