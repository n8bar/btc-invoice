Phase 4 now stays narrow: finish the RC-specific mail-readiness work only after the truthful notification model and active notification UX are already shipped and browser-verified in Phase 3.

## 1. Mailgun webhook integration
Wire Mailgun delivery events back into `invoice_deliveries` so delivery status is truthful and repeated failure patterns are detectable. This is the data foundation for the monitoring visibility planned in MS17's support UI.

1. [x] Add a signed Mailgun webhook endpoint (POST) that verifies the Mailgun HMAC signature before processing any event.
2. [x] Handle `delivered` events: find the matching `invoice_deliveries` row by Mailgun message ID and mark it `sent` with `sent_at`.
3. [x] Handle `failed` and `bounced` events: mark the matching row `failed` with the Mailgun reason in `error_message`. Log at `warning` level — these are actionable signals.
4. [x] Store the Mailgun message ID on the delivery row at send time so webhook events can find it.
   - This requires a new `provider_message_id` column on `invoice_deliveries` and capturing the ID from Laravel's `Mail::to()->send()` response.
5. [x] Protect the endpoint: reject requests with invalid signatures with a 403; return 200 for unknown message IDs (idempotent no-op, not an error Mailgun should retry).
6. [x] Add feature tests covering signature verification, `delivered` status update, and `failed` status update.
