Phase 4 now stays narrow: finish the RC-specific mail-readiness work only after the truthful notification model and active notification UX are already shipped and browser-verified in Phase 3.

## 1. Mailgun webhook integration
Wire Mailgun delivery events back into `invoice_deliveries` so delivery status is truthful and repeated failure patterns are detectable. This is the data foundation for MS17 alerting.

1. [ ] Add a signed Mailgun webhook endpoint (POST) that verifies the Mailgun HMAC signature before processing any event.
2. [ ] Handle `delivered` events: find the matching `invoice_deliveries` row by Mailgun message ID and mark it `sent` with `sent_at`.
3. [ ] Handle `failed` and `bounced` events: mark the matching row `failed` with the Mailgun reason in `error_message`. Log at `warning` level — these are actionable signals.
4. [ ] Store the Mailgun message ID on the delivery row at send time so webhook events can find it.
   - This requires a new `provider_message_id` column on `invoice_deliveries` and capturing the ID from Laravel's `Mail::to()->send()` response.
5. [ ] Protect the endpoint: reject requests with invalid signatures with a 403; return 200 for unknown message IDs (idempotent no-op, not an error Mailgun should retry).
6. [ ] Add feature tests covering signature verification, `delivered` status update, and `failed` status update.

## 2. Rehearse RC readiness
1. [ ] Validate the alias-on versus alias-off workflow explicitly.
2. [ ] Update rollout/contributor docs so pre-production safety and RC real-recipient verification are both clear.

3. [ ] Confirm the final transport path, public-link host behavior, and outbound-mail settings are ready for RC rollout.

## 3. Verify Phase 4
1. [ ] Mailgun webhook receives events, verifies signatures, and updates delivery rows correctly.
2. [ ] A `failed` delivery event is detectable in the `invoice_deliveries` table and logged at warning level.
3. [ ] RC mail-readiness guidance reflects the chosen transport and aliasing posture.
4. [ ] Alias-off rehearsal, public-link host behavior, and outbound-mail settings are verified without reopening the finished Phase 3 notification UX/truthfulness scope.
