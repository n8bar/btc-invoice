# MS16 Phase 1 Strategy - Delivery Baseline Audit

Status: Complete as of 2026-03-27. The live outbound inventory, current safeguards, and Phase 2 / Phase 3 inputs are now documented.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md), current delivery/alert code under `app/` and `tests/`

Phase 1 locks the trustworthy outbound-mail baseline for MS16 before any provider recovery work or broader notification-surface polish.

## 1. Reconstruct the live outbound surface
1. [x] Inventory every delivery type the app can currently queue and map each one to one of; use the live code, not older milestone notes, as the baseline inventory:
   - canonical in spec/product scope: `send`, `receipt`, `owner_paid_notice`, `past_due_owner`, `past_due_client`
   - still in RC scope but not yet fully specified: `client_overpay_alert`, `owner_overpay_alert`, `client_underpay_alert`, `owner_underpay_alert`
   - stale drift to retire or reshape: `client_partial_warning`, `owner_partial_warning` remain live even though the trimmed notifications spec no longer treats repeated partial payments as a settled client-facing alert family
   1. [x] Parallel: inventory mailable classes, delivery types, and tests under `app/Mail`, `app/Jobs`, `app/Services`, and `tests/Feature/*Notification*`.
      - Current mailables mirror the live delivery types in `DeliverInvoiceMail`.
      - Regression coverage exists for queueing the live delivery families, but only partial-warning resend behavior currently has an explicit cooldown test.
2. [x] Capture the current outbound surface explicitly:
   - manual invoice send (`send`) is queued from `InvoiceDeliveryController::store`
   - paid receipt (`receipt`) and owner paid notice (`owner_paid_notice`) are queued from the `InvoicePaid` event via `SendInvoiceReceipt`
   - overpay / underpay / partial-warning alerts are queued from `InvoiceAlertService::checkPaymentThresholds()`, which is called by watcher sync, manual adjustment refresh, and payment-correction refresh
   - past-due alerts are queued from the scheduled `invoices:send-past-due-alerts` command, which runs daily at `02:00`

## 2. Identify the stop-the-bleed risks
1. [x] Trace where deliveries are created, deduped, cooled down, skipped, and retried today.
   - delivery rows are created in three places only: `InvoiceDeliveryController::store`, `SendInvoiceReceipt`, and `InvoiceAlertService::queueDelivery`
   - delivery dispatch is centralized in `DeliverInvoiceMail`
   - watcher churn is throttled per invoice in `InvoicePaymentSyncService`, but that throttle does not act as mail-intent dedupe
   - queued truth changes are handled partly in `InvoiceAlertService::skipInvalidQueuedDeliveries()` and partly in `DeliverInvoiceMail::shouldSkipDelivery()`
2. [x] Identify the runaway/spam-prone path concretely enough to say:
   - no obvious infinite queue loop is present in the current job / event wiring
   - the real spam risk is the absence of one shared mail-intent gate: each path queues deliveries directly and applies its own narrow rules
   - the weakest path is manual send, which can queue unlimited duplicate `send` deliveries with no cooldown, dedupe, or explicit follow-up class
   - the next riskiest paths are payment-threshold refreshes (`watcher`, manual adjustment, payment correction), which all call `checkPaymentThresholds()` directly and can legitimately re-queue alerts once their local timestamps allow it
   - the daily past-due scheduler is bounded by timestamps, but it still relies on local per-type cooldown state rather than a shared delivery-intent policy
3. [x] Document which current safeguards already exist and which are too narrow:
   - existing safeguards:
     - one-off `exists()` checks on receipts / owner paid notices
     - per-invoice timestamp cooldowns for past-due, overpay, underpay, and partial-warning alerts
     - queued-delivery skip logic for paid, underpay, partial-warning, and past-due cases when invoice truth changes before send
   - too narrow or missing:
     - no cooldown / dedupe guard for manual `send`
     - no shared intent service across manual send, watcher-triggered alerts, scheduler-triggered alerts, and correction-triggered refreshes
     - no explicit transport-side circuit breaker or operator send-disable path
     - queued overpayment alerts have no skip path if the invoice state changes before send
     - queued manual sends do not re-check public-share eligibility before send
   1. [x] Parallel: collect a concise matrix of current cooldown fields and skip logic from `app/Models/Invoice.php`, `app/Services/InvoiceAlertService.php`, and correction-related tests without changing scope.
      - `receipt` / `owner_paid_notice`: dedupe via queued-or-sent existence checks; skip if invoice no longer `paid`
      - `past_due_owner` / `past_due_client`: 24h cooldown via `last_past_due_*`; skip if invoice becomes `paid` or `void`
      - `client_underpay_alert` / `owner_underpay_alert`: 24h cooldown via `last_underpayment_alert_at`; skip if underpayment resolves
      - `client_overpay_alert` / `owner_overpay_alert`: 24h cooldown via `last_overpayment_alert_at`; no skip-if-resolved path exists today
      - `client_partial_warning` / `owner_partial_warning`: 24h cooldown via `last_partial_warning_sent_at`; skip if warning no longer applies
4. [x] Flag current drift that later phases must resolve instead of inheriting silently:
   - manual sends currently have no shared cooldown/dedupe guard
   - current partial-payment warning mail is broader than the trimmed notifications spec
   - issuer-copy behavior is implemented today as separate owner/client alert classes rather than one explicit default issuer-copy rule plus toggle
   - test coverage proves only the partial-warning cooldown explicitly; equivalent cooldown coverage does not yet exist for overpay, underpay, or past-due alerts

## 3. Lock the Phase 2 and Phase 3 inputs
1. [x] Produce the minimum safeguard list that later work must implement before broader mail polish:
   - shared dedupe / notification-intent gating at delivery creation time
   - explicit policy for manual-send cooldowns versus intentional follow-up sends
   - pre-send revalidation for any queued delivery whose truth can drift before send, including manual sends and overpayment alerts
   - transport-side circuit-breaker or operator send-disable behavior
   - structured logging / evidence by delivery class and outcome so provider pressure is visible
2. [x] Produce the provider-recovery inputs needed for Phase 2:
   - what evidence is needed from Mailgun/sendability state: delivery logs, suppression/bounce state, domain auth/reputation status, and any recent provider complaints or throttles
   - whether aliasing currently hides or distorts that evidence: yes, aliasing protects recipients but can blur real-recipient validation and provider-side signal quality
   - what blocking constraints, if any, would prevent adopting Mailgun HTTP API in implementation: current repo constraints are implementation work, not framework blockers; the repo still needs Mailgun transport packages plus `config/services.php` / `config/mail.php` wiring
3. [x] Produce the truthfulness/rationalization inputs needed for Phase 3:
   - whether repeated partial-payment warnings survive, narrow, or retire
   - whether owner copies remain explicit mail classes, become issuer-copy behavior, or split by notice class
   - which payment-triggered sends may need owner review after ambiguity/correction scenarios
   - whether automatic receipts stay automatic or move behind an owner-reviewed acknowledgment / receipt split

Do not treat provider-account investigation as a Phase 1 sidecar if the next blocking decision depends on it; that work belongs to Phase 2.
