# MS17 Phase 5 Strategy - Post-Payment Onboarding

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 3 (off-chain payments) complete.
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

The existing getting-started flow (MS13 task 11) has three steps: wallet → invoice → deliver. It completes when the issuer has sent their first invoice. This phase adds Part 2: a fourth step that activates once a paid invoice is receipt-eligible and completes when the issuer sends the first reviewed client receipt.

Part 2 must respect the same rules as the rest of the receipt model: it suspends when ignore/reattribution review would make a receipt untruthful, and it does not auto-send anything.

## 1. Add the receipt step to GettingStartedFlow

1. [ ] Add `STEP_RECEIPT = 'receipt'` constant and include it in `stepOrder()` after `STEP_DELIVER`.
2. [ ] `receiptStepComplete()` — returns true when the issuer has at least one `invoice_deliveries` row with `type = 'receipt'` and `status` in `['queued', 'sent']` on a paid invoice.
3. [ ] `receiptStepActive()` — returns true when there is at least one paid invoice that is receipt-eligible (has no queued/sent receipt, no pending ignore/reattribution review blocking truthfulness). This gate determines whether Part 2 is shown at all; issuers who haven't had a paid invoice yet don't see the step.
4. [ ] Update `snapshot()` to include the receipt step only when `receiptStepActive()` is true, so Part 1 completions that predate any paid invoice don't surface a dangling step.
5. [ ] Update `stepDefinitions()` with label, title, body, and CTA for the receipt step. CTA links to the invoice show page of the qualifying paid invoice.
6. [ ] Update route definition in `web.php` to allow `receipt` as a valid step value alongside `wallet|invoice|deliver`.

## 2. Surface Part 2 in the UI

1. [ ] The getting-started progress strip on the invoice show page should appear when an issuer reaches a paid invoice from the flow, pointing them to the receipt review panel.
2. [ ] The dashboard "Receipts waiting for review" prompt is already present — confirm it links naturally into the getting-started receipt step when the flow is active.
3. [ ] Part 2 completes silently when `receiptStepComplete()` becomes true; no separate completion screen needed.

## 3. Verify Phase 5

1. [ ] An issuer with no paid invoices does not see the receipt step in the getting-started flow.
2. [ ] Once a paid invoice is receipt-eligible, the receipt step appears and the flow resumes.
3. [ ] The step suspends (does not show as completable) when ignore/reattribution review would make a receipt untruthful.
4. [ ] Sending the first reviewed client receipt marks the step complete.
5. [ ] An issuer paid via off-chain payment (Phase 3) also triggers Part 2 correctly.
6. [ ] Feature tests cover the step activation, suspension, and completion conditions.
