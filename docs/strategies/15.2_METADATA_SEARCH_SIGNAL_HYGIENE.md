# MS15 Phase 2 Strategy - Metadata + Search-Signal Hygiene

Status: Draft.
Parent milestone doc: [`docs/milestones/15_CRYPTOZING_APP_SEO_BOOTSTRAP.md`](../milestones/15_CRYPTOZING_APP_SEO_BOOTSTRAP.md)

Tighten the placeholder's technical search signals and only the minimum on-page copy that Phase 1 proves needs adjustment.

Owner tags for unchecked work: `Agent` = critical-path repo/live-host/docs work I should drive. `Subagent` = safe delegated sidecar work I can hand off once the earlier prerequisites in this phase are done. `User` = approval/scope judgment you should drive. `Shared` = I should prepare/recommend and you should confirm the content or scope call.

## 1. Lock the scope boundary before edits
1. [ ] [Shared] Confirm which Phase 1 findings are true metadata/search-signal issues versus broader marketing/content ideas.
2. [ ] [Shared] Keep this phase limited to titles, descriptions, headings, canonical/schema signals, image/meta references, and only the minimum on-page copy needed to support the existing landing page.
3. [ ] [Shared] Reject any proposed changes that amount to a landing-page rewrite, messaging overhaul, or new-page expansion.
4. [ ] [Shared] If Phase 1 says the current one-page/public Helpful Notes mix is too thin, allow derivative public Helpful Notes content under `site/` only to the extent needed to strengthen SEO while the site still reads as a product placeholder.

## 2. Correct repo-controlled metadata and discovery signals
1. [ ] [Subagent] Fix any incorrect or weak `<title>`, meta description, canonical, Open Graph, Twitter, or JSON-LD fields identified in Phase 1.
2. [ ] [Subagent] Verify social-preview assets, image URLs, and structured-data references remain accurate on the live placeholder host.
3. [ ] [Subagent] Tighten `site/robots.txt` and `site/sitemap.xml` only if Phase 1 proved a real search-signal problem.
4. [ ] [Shared] Decide whether any sitemap `lastmod` update process belongs in the repo, in workflow discipline, or in later launch operations; avoid fake automation.

## 3. Make only the minimum on-page copy adjustments needed
1. [ ] [Shared] Limit on-page copy changes to the minimum required to support accurate metadata/search signals or remove claims Phase 1 proved risky.
2. [ ] [Agent] Keep the existing placeholder positioning and structure intact.
3. [ ] [Shared] Re-check that the placeholder still reads as the same landing page after any copy touch.
4. [ ] [Shared] If derivative public Helpful Notes content is needed, add only as much as materially improves SEO coverage without turning the placeholder into a broad knowledge base.

## 4. Preserve later rollout constraints while editing
1. [ ] [Agent] Keep the root URL, canonical intent, and GitHub Pages deployment assumptions compatible with the later live landing-page replacement.
2. [ ] [Agent] Avoid introducing placeholder-only metadata or copy choices that the live landing page would immediately need to undo.
3. [ ] [Agent] Note any SEO-related changes that later rollout work must preserve at go-live.

## 5. Verify Phase 2
1. [ ] [Shared] Confirm every shipped change in this phase maps back to a Phase 1 finding or decision instead of open-ended content polishing.
2. [ ] [Subagent] Confirm the live placeholder still exposes the intended title, description, canonical, social, and JSON-LD signals after the edits.
3. [ ] [Shared] Confirm the placeholder content still feels like the same page, not a rewritten marketing surface.
4. [ ] [Agent] Confirm any later rollout constraints discovered here are captured at the right higher level instead of being rediscovered at deployment time.
5. [ ] [Shared] If derivative public Helpful Notes content shipped, confirm it improves SEO coverage while the site still reads as a product placeholder rather than a docs encyclopedia.
