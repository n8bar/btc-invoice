# MS16 Phase 2 Strategy - Safeguards + Provider Recovery

Status: Pending.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)
Depends on: [`docs/strategies/16.1_DELIVERY_BASELINE_AUDIT.md`](16.1_DELIVERY_BASELINE_AUDIT.md)
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md), [`docs/ops/RC_ROLLOUT_CHECKLIST.md`](../ops/RC_ROLLOUT_CHECKLIST.md)

Phase 2 restores a trustworthy delivery path by combining app-side safeguards with the provider/transport decision.

## 1. Confirm the provider state
1. [ ] Determine the actual Mailgun/domain sendability state and capture concrete evidence for throttling, blocking, or reputation damage.
2. [ ] If dashboard or provider-console access is required, treat that step as `Guided User` work and capture only the evidence needed to unblock the transport decision.
3. [ ] Decide whether temporary aliasing is still useful for this phase or whether it now distorts the validation signal too much.

## 2. Implement shared send safeguards
1. [ ] Add the shared guardrails Phase 1 proved necessary around delivery creation and dispatch.
2. [ ] Prefer one shared intent/dedupe layer over hand-coded per-notice checks that drift by mail class.
3. [ ] Ensure manual send, watcher-triggered mail, scheduler-triggered mail, and correction-sensitive queued sends all pass through the same safety policy where appropriate.
4. [ ] Add or tighten:
   - duplicate-send suppression
   - per-class cooldown enforcement
   - emergency send-disable / circuit-breaker behavior
   - clearer logging and delivery-failure evidence

## 3. Decide and ship the transport path
1. [ ] Evaluate staying on SMTP versus moving to the Mailgun HTTP API using the criteria locked in Phase 1.
2. [ ] Document the decision in the relevant docs and implement it if a transport change is required.
3. [ ] Keep the notifications spec transport-agnostic; implementation and ops docs hold the concrete transport choice.

## 4. Verification and recovery proof
1. [ ] Run controlled send tests against the chosen transport with the new safeguards enabled.
2. [ ] Confirm that duplicate-send prevention and send-disable behavior work without breaking legitimate delivery attempts.
3. [ ] Verify that queued deliveries still mark `skipped` or `failed` truthfully when invoice state changes or transport errors occur.

## 5. Safe parallel sidecars
- Safe sidecar: provider-state evidence gathering and RC rollout doc note updates, as long as the main implementation path is not blocked waiting for those artifacts.
- Safe sidecar: add/expand focused delivery tests once the main safeguard shape is locked.
- Avoid splitting work across overlapping delivery-creation files until the shared gating seam is chosen.

## 6. Verify Phase 2
1. [ ] Mailgun sendability is restored or an explicit alternate path is chosen.
2. [ ] Shared outbound-mail safeguards are implemented and documented.
3. [ ] The `SMTP` vs `Mailgun HTTP API` decision is explicit, evidence-based, and reflected in docs/code as needed.
