## 2. Implement shared send safeguards
1. [x] Add the shared guardrails Phase 1 proved necessary around delivery creation and dispatch.
2. [x] Prefer one shared intent/dedupe layer over hand-coded per-notice checks that drift by mail class.
3. [x] Ensure manual send, watcher-triggered mail, scheduler-triggered mail, and correction-sensitive queued sends all pass through the same safety policy where appropriate; avoid splitting work across overlapping delivery-creation files until the shared gating seam is chosen.
4. [x] Add or tighten:
   - duplicate-send suppression
   - per-class cooldown enforcement
   - emergency send-disable / circuit-breaker behavior
   - clearer logging and delivery-failure evidence
   1. [x] Parallel: add or expand focused delivery tests without reopening the gating design.
5. [ ] Add stronger outbound idempotency before any real provider send.
   1. [x] Make delivery creation idempotent at the queue-intent boundary so concurrent duplicate attempts cannot create multiple real outbound intents.
   2. [ ] Make send execution idempotent at the job/provider boundary so retries or duplicate job execution cannot produce a second provider send after the first has already effectively happened.
   3. [ ] Base those guards on stable business intent rather than rendered content:
      - invoice id
      - notice class
      - normalized recipient / recipient role
      - payment identity such as `txid` when the notice is payment-scoped
      - never on variable timestamps, provider-added headers, or rendered body drift
   4. [ ] Add focused tests for the new idempotency boundary so the first real Mailgun verification send is not the first proof.
      - Current state: the queue-intent boundary now has a focused locked-intent test; leave this item open until the send-boundary half has matching coverage too.

2. [x] Implement the standard Laravel Mailgun driver wiring.

   3. [x] Add the `mailgun` mailer definition in `config/mail.php`.

3. [x] No concrete blocking constraint has appeared so the phase stays on Mailgun HTTP API; if one does appear, document it explicitly instead of quietly drifting back to SMTP.
