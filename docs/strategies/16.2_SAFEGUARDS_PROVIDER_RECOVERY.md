# MS16 Phase 2 Strategy - Safeguards + Provider Recovery

Status: Active. Shared safeguards and Laravel Mailgun wiring are now in the app; provider-state confirmation and controlled transport verification are still pending.
Parent milestone: [`docs/milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md`](../milestones/16_MAILER_AND_ALERTS_POLISH_AUDIT.md)
Depends on: [`docs/strategies/16.1_DELIVERY_BASELINE_AUDIT.md`](16.1_DELIVERY_BASELINE_AUDIT.md)
Canonical inputs: [`docs/specs/NOTIFICATIONS.md`](../specs/NOTIFICATIONS.md), [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md), [`docs/ops/RC_ROLLOUT_CHECKLIST.md`](../ops/RC_ROLLOUT_CHECKLIST.md)

Phase 2 restores a trustworthy delivery path by combining app-side safeguards with provider recovery and the locked Mailgun HTTP API transport direction.

## 1. Confirm the provider state
1. [ ] Determine the actual Mailgun/domain sendability state and capture concrete evidence for throttling, blocking, or reputation damage. If dashboard or provider-console access is required, treat that step as `Guided User` work and capture only the evidence needed to confirm Mailgun HTTP API remains viable for production use.
2. [ ] Decide whether temporary aliasing is still useful for this phase or whether it now distorts the validation signal too much.
   1. [ ] Parallel: update RC rollout notes with only the provider/aliasing findings that are already settled enough to document.

## 2. Implement shared send safeguards
1. [x] Add the shared guardrails Phase 1 proved necessary around delivery creation and dispatch.
2. [x] Prefer one shared intent/dedupe layer over hand-coded per-notice checks that drift by mail class.
3. [x] Ensure manual send, watcher-triggered mail, scheduler-triggered mail, and correction-sensitive queued sends all pass through the same safety policy where appropriate; avoid splitting work across overlapping delivery-creation files until the shared gating seam is chosen.
4. [x] Add or tighten:
   - duplicate-send suppression
   - per-class cooldown enforcement
   - emergency send-disable / circuit-breaker behavior
   - clearer logging and delivery-failure evidence
   1. [x] Parallel: add or expand focused delivery tests without reopening the gating design.

## 3. Ship the Mailgun HTTP API path
1. [x] Treat Mailgun HTTP API as the intended MS16 transport; do not preserve SMTP by default, and keep the notifications spec transport-agnostic while implementation and ops docs carry the concrete transport choice.
2. [x] Implement the standard Laravel Mailgun driver wiring.
   1. [x] Add the Laravel Mailgun transport dependencies (`symfony/mailgun-mailer` and `symfony/http-client`).
   2. [x] Add the Mailgun credentials/config block in `config/services.php`.
   3. [x] Add the `mailgun` mailer definition in `config/mail.php`.
   4. [x] Set and document the intended `MAIL_MAILER=mailgun` path so the HTTP transport is the default send path instead of SMTP.
   5. [x] Update the relevant ops/env docs with the exact Mailgun variables and any region/endpoint requirements.
3. [x] No concrete blocking constraint has appeared so the phase stays on Mailgun HTTP API; if one does appear, document it explicitly instead of quietly drifting back to SMTP.

## 4. Verification and recovery proof
1. [ ] Run controlled send tests against the chosen transport with the new safeguards enabled.
2. [x] Confirm that duplicate-send prevention and send-disable behavior work without breaking legitimate delivery attempts.
3. [x] Verify that queued deliveries still mark `skipped` or `failed` truthfully when invoice state changes or transport errors occur.

## 5. Verify Phase 2
1. [ ] Mailgun sendability is restored or an explicit alternate path is chosen.
2. [ ] Shared outbound-mail safeguards are implemented and documented.
3. [ ] Mailgun HTTP API is explicit as the chosen transport and reflected in docs/code, or any blocking fallback is explicitly justified.
