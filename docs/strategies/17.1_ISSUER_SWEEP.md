# MS17 Phase 1 Strategy - "owner" → "issuer" Sweep

Status: In progress.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Canonical inputs: [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

Owner tags: `[Agent]` = critical-path code and doc work to drive directly. `[Subagent]` = safe delegated sidecar work that can run once prerequisites in this phase are done. `[User]` = work you can drive directly without specialized coaching.

The word "owner" is used throughout the codebase to mean the person who creates and sends invoices — the invoice issuer. This phase replaces it everywhere: UI copy, URLs, route names, code variables and method names, mail class names, mail template files, delivery type strings, database columns, and tests. The changelog is the only exception — historical entries are left as-is since they are a record of what was said at the time.

Two changes touch the database and require migrations:
- `invoices.last_past_due_owner_alert_at` column rename
- `invoice_deliveries.type` values that contain "owner" (stored string data)

Do these last, after all code changes are in place, so the migration and the code land together.

## Delivery type string rename map

These string values are stored in `invoice_deliveries.type` and referenced across services, jobs, and models. Every occurrence of the old value must be replaced with the new one in code before the migration runs.

| Old | New |
|---|---|
| `owner_paid_notice` | `issuer_paid_notice` |
| `payment_acknowledgment_owner` | `payment_acknowledgment_issuer` |
| `past_due_owner` | `past_due_issuer` |
| `owner_overpay_alert` | `issuer_overpay_alert` |
| `owner_underpay_alert` | `issuer_underpay_alert` |
| `owner_partial_warning` | `issuer_partial_warning` |
