# MS17 Phase 2 Strategy - Test Suite Rationalization

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`tests/Feature/`](../../tests/Feature/), [`tests/Unit/`](../../tests/Unit/)

The test suite has grown large and was written opportunistically rather than intentionally. Many tests may be green but irrelevant, outdated, or testing implementation details rather than product behavior. This phase audits the suite, produces a written recommendation for each test file, gets approval, and executes the agreed changes so the suite is intentional and trustworthy going into RC.

## 1. Audit the test suite

Produce [`docs/qa/tests/TEST_RATIONALIZATION_AUDIT.md`](../qa/tests/TEST_RATIONALIZATION_AUDIT.md) — one entry per test file covering: what it tests, whether it still reflects current product behavior, and a recommendation.

Recommendations use one of four options:
- **Keep** — tests are relevant and correct as-is.
- **Tweak** — tests are worth keeping but need adjustment (too broad, too narrow, asserting the wrong thing, or drifted from current behavior).
- **Merge** — two or more tests are covering the same scenario; combine into one cleaner test.
- **Delete** — tests are irrelevant, outdated, or provide no meaningful signal.

Rate each file on two axes before recommending:
- **Relevance** — does this test cover behavior that still exists and matters for RC?
- **Correctness** — is the test asserting the right things, or just asserting something?

1. [ ] List all test files and group by feature area.
2. [ ] For each file, read the tests, rate relevance and correctness, and write a recommendation with reasoning.
3. [ ] Flag any surfaces with zero coverage that matter for RC.
4. [ ] Publish the completed audit doc for review.

## 2. Review recommendations

Walk through the audit doc together. Adjust any recommendations before execution begins. No changes to the test suite happen until this review is complete.

1. [ ] Review audit doc findings together.
2. [ ] Agree on final recommendations for each file.
3. [ ] Note any new tests to add that the audit surfaces as gaps.

## 3. Execute approved changes

1. [ ] Delete approved files/tests.
2. [ ] Tweak or merge approved files/tests.
3. [ ] Add any new tests agreed in review.
4. [ ] Confirm full suite passes cleanly after changes.

## 4. Verify Phase 2

1. [ ] Test suite passes with no failures.
2. [ ] No test was changed without an approved recommendation in the audit doc.
3. [ ] Any identified coverage gaps are either filled or explicitly deferred to backlog.
4. [ ] Suite runtime is improved.
