# MS17 Phase 2 Strategy - Test Suite Rationalization

Status: In progress.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`tests/Feature/`](../../tests/Feature/), [`tests/Unit/`](../../tests/Unit/)

The test suite has grown large and was written opportunistically rather than intentionally. This phase runs in two tracks:

- **Track A — Rationalization**: Reduce the number of tests, eliminate redundancy, and shorten the suite runtime. The goal is a leaner suite that still covers all RC-critical behavior.
- **Track B — Gap closure**: Plan and execute tests for the RC-critical surfaces the audit identified as having no meaningful coverage.

These tracks are ordered: Track A first (less to maintain before adding new tests), Track B second.

## 2. Pass 2 — Efficiency and redundancy audit

A second read of the suite with fresh eyes, this time looking at *how* the tests are written rather than *what* they cover. Goal: produce a written set of efficiency recommendations before touching any code.

Focus areas:

- **Runtime profiling** — run the suite with per-test timing and identify the slowest tests and files. Flag any test that takes > 1s for scrutiny.
- **Cross-file redundancy** — identify scenarios tested in more than one file. Flag for consolidation or deletion of the duplicate.
- **Setup overhead** — look at `makeInvoice` and other helpers duplicated across files. Identify shared helpers or traits that could replace them. Flag `RefreshDatabase` vs. `DatabaseTransactions` opportunities.
- **Over-asserting** — flag tests that check incidental output (raw HTML snippets, route names, unrelated sidebar copy) when a single behavioral assertion would suffice.
- **Under-asserting** — flag tests that only assert HTTP status when the actual behavioral outcome (DB state, redirect target, session key) is more meaningful and no harder to write.

Deliverable: append an `## Efficiency Recommendations` section to `TEST_RATIONALIZATION_AUDIT.md` with specific, actionable items before any code changes.

## 4. Plan Track B — Gap closure

All five gaps identified in Pass 1 are filling now. Test specs are written inline below — review and adjust before execution begins.

### Gap 1 — Blockchain confirmation logic

`WatchPaymentsCommandTest` was only partially read during the audit. Read the full file first; fill only what is genuinely missing after confirming what already exists.

Scenarios to cover:
- Payment detected but not yet confirmed — invoice stays in `partial`/`sent`, no paid transition
- Payment reaches confirmation threshold — invoice transitions to `paid`
- Mempool API returns an error — command logs and continues without crashing
- Payment already recorded is not duplicated on a second run

1. [ ] [Agent] Read `WatchPaymentsCommandTest` in full and record what is already covered.
2. [ ] [Agent] Write tests only for scenarios confirmed missing.

### Gap 2 — Receipt truthfulness invariants

Scenarios to cover:
- Receipt send is blocked when the invoice has an unresolved ignored payment
- Receipt send is blocked when a reattribution is pending
- Receipt send proceeds normally when no correction state exists

1. [ ] [Agent] Write tests for receipt blocking conditions in a new `ReceiptTruthfulnessTest` or append to `InvoicePaymentCorrectionTest`.

### Gap 3 — Invoice state machine

Scenarios to cover:
- `draft → sent` via the send action
- `sent → partial` on first underpayment detection
- `partial → paid` when balance is resolved
- `sent → paid` on full payment in one transaction
- Transitions are blocked when the invoice is `void` or `trashed`

1. [ ] [Agent] Write state machine transition tests. Append to `InvoiceShowEditFlowTest` or create `InvoiceStatusTransitionTest` if the file would become unwieldy.

### Gap 4 — Multi-wallet / wallet reconnect

Scenarios to cover:
- Issuer reconnects a new wallet — existing invoices retain their original lineage
- Address derivation uses the current wallet key for new invoices, not the retired key
- Watcher uses invoice lineage even when the current wallet key differs (partially covered per audit — confirm and fill gaps)

1. [ ] [Agent] Read `WatchPaymentsCommandTest` in full (per Gap 1) — confirm lineage tests already present before writing new ones.
2. [ ] [Agent] Write any missing reconnect/lineage scenarios.

### Gap 5 — Dashboard cache invalidation on payment events

Scenarios to cover:
- Cache is invalidated when a payment is recorded on an invoice belonging to the user
- Stale cache does not persist past a payment event (i.e., dashboard reflects new state on next load)
