# MS17 Phase 4 Strategy - Support UI + Monitoring

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md)

The support UI is already complete: dashboard lists active support grants, invoice index and detail (with payment history and deliveries), and client index and detail all exist and are read-only gated by `isSupportAgent()` + `hasActiveSupportAccessGrant()`. No gaps to fill there.

This phase adds a service health monitoring panel to the support dashboard so a support agent can triage operational issues — queue depth, recent delivery failures, watcher health — without raw log access.

## 1. Add service health monitoring to the support dashboard

1. [ ] Queue depth — count of `invoice_deliveries` rows with `status = 'queued'` or `status = 'sending'`.
2. [ ] Recent delivery failures — count and list of `invoice_deliveries` rows with `status = 'failed'` in the last 24 hours, showing delivery type, recipient, and `error_message`.
3. [ ] Watcher health — surface the last `created_at` timestamp of any `invoice_payments` row as a proxy for last watcher activity. If no payment in the last N minutes, flag as stale.
4. [ ] Render the monitoring panel at the top of the support dashboard view, above the issuer grants table.

## 2. Verify Phase 4

1. [ ] Monitoring panel shows queue depth, recent delivery failures, and watcher health on the support dashboard.
2. [ ] Delivery failure rows show type, recipient, and error message.
3. [ ] Watcher health indicator correctly reflects recent activity or staleness.
4. [ ] No write or admin actions are introduced.
5. [ ] Feature test covers the monitoring panel data rendering.
