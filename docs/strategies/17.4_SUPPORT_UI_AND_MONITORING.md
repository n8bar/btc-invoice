# MS17 Phase 4 Strategy - Support UI + Monitoring

Status: Pending.
Parent milestone: [`docs/milestones/17_PRODUCT_READINESS.md`](../milestones/17_PRODUCT_READINESS.md)
Depends on: Phase 1 ("owner" → "issuer" sweep) complete.
Canonical inputs: [`docs/PRODUCT_SPEC.md`](../PRODUCT_SPEC.md), [`routes/web.php`](../../routes/web.php)

The support UI has basic read-only access to issuers, their invoices, and their clients. This phase fills in whatever gaps remain for RC and adds a service health monitoring surface to the support dashboard so a support agent can triage operational issues — queue depth, recent delivery failures, watcher health — without raw log access.

## 1. Audit the current support UI

Identify what is already in place and what is missing for RC.

1. [ ] Review the support dashboard, invoice, and client views for gaps — missing data, broken layouts, or surfaces a support agent would need that aren't there.
2. [ ] List any issuer or invoice data that is accessible in the app but not surfaced in the support UI.
3. [ ] Document findings before making any changes.

## 2. Fill support UI gaps

Address any gaps identified in section 1. Scope is limited to RC-necessary read access — no write or admin actions.

1. [ ] Implement agreed gap fixes.
2. [ ] Confirm the support agent flow covers: finding an issuer, reviewing their invoices and clients, and reading delivery history.

## 3. Add service health monitoring to the support dashboard

Add a monitoring panel to the support dashboard that surfaces operational health signals without requiring log access.

1. [ ] Queue depth — show the current count of queued and in-flight `invoice_deliveries` rows.
2. [ ] Recent delivery failures — show a count and list of `invoice_deliveries` rows with `status = 'failed'` in the last 24 hours, with delivery type, recipient, and error message.
3. [ ] Watcher health — show the last time the `wallet:watch-payments` command completed successfully (derive from a lightweight heartbeat or last-processed timestamp).
4. [ ] Keep the monitoring panel read-only and scoped to the support agent role — no actions, no raw logs.

## 4. Verify Phase 4

1. [ ] Support agent can find an issuer, view their invoices and clients, and read delivery history without hitting a dead end.
2. [ ] Monitoring panel shows queue depth, recent delivery failures, and watcher health on the support dashboard.
3. [ ] Delivery failure rows show enough context (type, recipient, error) to triage without log access.
4. [ ] No write or admin actions are accessible from the support UI.
5. [ ] Feature tests cover the support dashboard monitoring panel data.
