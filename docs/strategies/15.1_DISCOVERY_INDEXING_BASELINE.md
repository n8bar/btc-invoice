## 2. Tighten the repo-controlled discovery signals
1. [x] Verify the canonical, Open Graph, Twitter, JSON-LD, and image URLs resolve correctly on the live placeholder host.
   - Current result on 2026-03-26: `https://cryptozing.app/` serves the expected canonical URL, `og:url`, `og:image`, `twitter:image`, and JSON-LD on the live GitHub Pages host. `https://cryptozing.app/og-preview.png` and `https://cryptozing.app/images/CZ.png` both resolve successfully.
2. [x] Decide whether the sitemap should remain single-URL or include any additional crawlable placeholder URLs that genuinely exist; do not invent URLs just for SEO.
   - Current decision on 2026-03-26: keep `site/sitemap.xml` single-URL for now because the repo-controlled GitHub Pages placeholder currently only exposes `/` as real crawlable placeholder content. Public Helpful Notes currently lives on the Laravel app surface, not under `site/`, so it should not be added to the placeholder sitemap unless a later MS15 phase deliberately publishes derivative Helpful Notes content there.
3. [x] Tighten any missing or incorrect technical discovery signals found in the audit without reopening broad copy work.
   - Current result on 2026-03-26: no additional repo-controlled discovery-signal fixes are needed right now. The live placeholder host matches the intended root URL, metadata, image references, robots file, and sitemap behavior already present in `site/`.
4. [x] Decide how `site/sitemap.xml` `lastmod` should be maintained so it reflects meaningful placeholder changes.
   - Current decision on 2026-03-26: maintain `site/sitemap.xml` `lastmod` manually in the same commit whenever the placeholder's root-page content, metadata, social-preview asset, or other crawl-visible `site/` content changes meaningfully. Do not bump it for unrelated repo work.
5. [x] Check that the current robots/sitemap/canonical setup will still make sense when the live landing page replaces the placeholder at `/`.
   - Current result on 2026-03-26: yes. The current setup still makes sense as long as the live app landing page keeps the same root URL and canonical intent, continues serving crawl-allowing robots behavior plus a root sitemap, and preserves the SEO baseline handoff already noted for later deployment work.
